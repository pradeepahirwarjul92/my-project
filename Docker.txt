03. Docker Compose
Ans - Running multiple containers manually is messy
		-	Spring Boot app
		-	MySQL database
		-	Redis cache
		-	Nginx reverse proxy
Without Docker Compose, you’d have to run each container manually:	
docker run -d --name mysql -e MYSQL_ROOT_PASSWORD=root mysql:8
docker run -d --name redis redis:latest
docker run -d --name app --link mysql:mysql --link redis:redis app-image
docker run -d --name nginx -p 80:80 nginx

You have to remember all the ports, environment variables, container links.
If you restart, you have to run them in the right order.
Hard to share with colleagues because they’d have to repeat all commands.

⚠️ This quickly becomes error-prone as projects grow.
	
-	 Docker Compose solves this
	Docker Compose allows you to define all services in a single YAML file, including:

		Which containers to run (services)
		Environment variables (environment)
		Ports (ports)
		Volumes (volumes)
		Start order (depends_on)
		Networks (networks)
Example : services:
  mysqldb:
    image: mysql:8.0
    container_name: mysqldb
    restart: always
    environment:
      MYSQL_ROOT_PASSWORD: root
      MYSQL_DATABASE: user_db
    ports:
      - "3307:3306"
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      interval: 10s
      timeout: 5s
      retries: 10

  user-service:
    build: .
    container_name: user-service
    ports:
      - "9099:9099"
    depends_on:
      mysqldb:
        condition: service_healthy
    environment:
      SPRING_DATASOURCE_URL: jdbc:mysql://mysqldb:3306/user_db?allowPublicKeyRetrieval=true&useSSL=false&serverTimezone=UTC
      SPRING_DATASOURCE_USERNAME: root
      SPRING_DATASOURCE_PASSWORD: root		
	  
	 docker-compose up -d
	
All containers start correctly, in the right order, with correct configuration.	

3️ Benefits in real-world projects
	Easy setup: One file describes your entire system. No manual commands.
	Reproducibility: Colleagues or CI/CD systems can run the same setup anywhere.
	Persistent data: Volumes ensure your database doesn’t lose data when containers restart.
	Environment flexibility: Use .env files to switch dev, test, or prod environments.
	
	Scaling services: You can run multiple instances of a service easily:
	docker-compose up --scale app=3
	
	Cleaner workflow: You can stop all services at once:
	docker-compose down

1️⃣ Run MySQL in Docker
You need a running MySQL container where your Spring Boot app can connect.

docker run --name user-mysql \
  -e MYSQL_ROOT_PASSWORD=root \
  -e MYSQL_DATABASE=user_db \
  -p 3306:3306 \
  -v mysql-data:/var/lib/mysql \
  -d mysql:8
Explanation:

--name user-mysql → container name

MYSQL_ROOT_PASSWORD=root → MySQL root password

MYSQL_DATABASE=user_db → your database

-p 3306:3306 → map MySQL port to host

-v mysql-data:/var/lib/mysql → persist data even if container stops

Check if it’s running:

docker ps
2️⃣ Run Your Spring Boot App
Assuming you already built your Docker image (docker-demo:1.0), run it:

docker run --name user-service \
  -p 9099:9099 \
  --link user-mysql:mysql \
  -d docker-demo:1.0
Explanation:

--link user-mysql:mysql → connects your app to MySQL container

-p 9099:9099 → maps app port to host

Now your app can access MySQL at jdbc:mysql://mysql:3306/user_db

Perfect! Now that you have a working Docker Compose setup, the next move is to really understand how Docker Compose works, how to customize it, and use it in real-world projects. I’ll lay out a clear learning path with practical steps:

1️⃣ Understand the Basics Deeply
Even if you’ve used Compose once, make sure you know why each part exists:

services: – each container your project needs (app, database, cache, etc.)

volumes: – persist data outside the container (like your MySQL database)

ports: – expose container ports to your host machine

depends_on: – order startup of containers

environment: – pass environment variables to containers

networks: – how containers talk to each other

💡 Tip: Try changing ports, environment variables, and see how it affects your app.

2️⃣ Practice With Multiple Services
Create projects with 2–3 services:

Spring Boot + MySQL + Redis

Node.js + MongoDB + Nginx

Python Flask + PostgreSQL

Practice connecting them via Compose, exposing ports, and sharing data via volumes.

3️⃣ Learn Compose Commands
The most common commands:

Command	What it does
docker-compose up	Build/start all containers
docker-compose up -d	Start in background (detached)
docker-compose down	Stop and remove containers, networks, volumes
docker-compose logs -f	Stream logs of all services
docker-compose exec <service> bash	Access container terminal
docker-compose ps	List running services
💡 Tip: Try logs -f while your app starts, you’ll see startup order and errors clearly.

4️⃣ Use .env Files
Instead of hardcoding passwords or ports in docker-compose.yml, you can use a .env file:

MYSQL_ROOT_PASSWORD=root
MYSQL_DATABASE=user_db
APP_PORT=9099
Then in Compose:

environment:
  MYSQL_ROOT_PASSWORD: ${MYSQL_ROOT_PASSWORD}
  MYSQL_DATABASE: ${MYSQL_DATABASE}
ports:
  - "${APP_PORT}:${APP_PORT}"
✅ This is industry standard and makes your setup flexible.

5️⃣ Experiment With Advanced Features
Networks: isolate frontend/backend databases

Healthchecks: ensure app waits until DB is ready

Build vs Image: use build: ./app to build Dockerfile dynamically

Override files: docker-compose.override.yml for dev vs prod

Scaling: docker-compose up --scale app=3 to run multiple app containers

6️⃣ Hands-On Projects

Make MySQL and Spring Boot communicate via network

Add automated DB seed scripts

Use volumes and test persistence

Introduce a caching layer (Redis) and watch service communication

7️⃣ Learning Resources
Official docs: https://docs.docker.com/compose/

Tutorials: search for “Docker Compose real projects”

YouTube hands-on examples: Spring Boot + MySQL + Docker Compose
