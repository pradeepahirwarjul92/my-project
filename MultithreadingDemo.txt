🔰 STEP 1 — Basics (Must Know First)

✅ 1. Thread vs Process
Ans - 
	Process :
		Independent program
		Own memory (heap + stack)
		Heavyweight
		Example: Chrome, IntelliJ, MySQL server
		
	Thread :
		Lightweight execution unit inside a process
		Shares memory with other threads
		Faster communication
	👉 One process → many threads

✅ 2. Thread vs Runnable (Most Asked)
Ans -
		Feature					Thread					Runnable
		Task + Execution		Combined				Separated
		Inheritance				Uses Thread class		No inheritance needed
		Multiple inheritance	❌ No					✅ Yes
		ExecutorService			❌ Poor support			✅ Designed for it
		
👉 Preferred: Runnable
👉 Because it separates what to do from how to run		


✅ 3. start() vs run()
Ans -
	start()
			- Creates new thread
			- Calls run() internally
			- Executes concurrently
	run()
			- Normal method call
			- Runs on current thread			
	👉 Interview line:
			- start() → new thread
			- run() → same thread

✅ 4. Can a thread be restarted?
Ans -
	❌ NO
	Thread lifecycle is fixed.
	Once TERMINATED → cannot start again.
	start() again → IllegalThreadStateException
	
👉 Create new Thread object instead

✅ 5. Thread Lifecycle
Ans -
	NEW → RUNNABLE → RUNNING → BLOCKED/WAITING → TERMINATED

Example:
		Create thread → NEW
		start() → RUNNABLE
		CPU executes → RUNNING
		wait()/sleep()/lock → BLOCKED/WAITING
		finish → TERMINATED

✅ 6. Context Switching
Ans -
		CPU switches between threads.

	🔴 Costly because:
		Save thread state
		Load another state
	No real work during switch
👉 Too many threads = performance drop

✅ 7. Daemon Thread
Ans -
	Background support thread.
		✔ JVM stops daemon threads automatically when user threads finish

	Examples:
		Garbage Collector
		Logging
		Cache cleanup
		Monitoring
		Session timeout checker

✅ 8. Thread Scheduling
Ans -
	Handled by OS, not JVM.
	✔ Priorities are only hints
	✔ No execution order guarantee

✅ 9. Race Condition
Ans -
	Occurs when:
		👉 Multiple threads modify shared data without synchronization

Example:
		counter++   // not atomic
		Production example:
		Bank balance update
		Inventory count
		Ticket booking system

Code : 
	class BankAccount{
		private int balance = 1000; 
		
		public void withdraw(int amount) {
			if(balance >= amount){ // step 1 : check
				try{
					Thread.sleep(100);
				}catch(Exception e){
				}
				balance -=amount;
			}
		}
		
		public double getBalance(){
		return balance;
		}
	
	}
	
	public class Demo{
		public static void main(String args[]) throws Exception{
		
			BankAccount acc= new BankAccount();
			Thread t1 = new Thread(() -> acc.withdraw(700));
			Thread t2 = new Thread(() -> acc.withdraw(700));
			
			t1.start();
			t2.start();
			
			t1.join();
			t2.join();
			
			System.out.println("Final Balance : " +acc.getBalance());
		}
	}
		
	💥 What Happens Internally

		Initial balance = 1000
		T1 checks → balance >= 700 → TRUE
		T2 checks → balance >= 700 → TRUE

		T1 deducts → balance = 300
		T2 deducts → balance = -400   ❌

👉 Impossible state created
👉 Real financial corruption	

🎯 Interview One-Line Killer
	Race condition occurs because check-then-act is not atomic.

✅ 11. Object-level vs Class-level Lock
Ans -
	Object Lock

		Lock on instance (this)

		Different objects → no blocking

synchronized void withdraw() {}
Class Lock

Lock on Class object

Blocks all instances

static synchronized void audit() {}
✅ 12. Can two synchronized methods run on same object?

❌ NO

Because both need the same object lock.

✅ 13. Deadlock

Two threads waiting for each other forever.

T1 → lock A → wait for B  
T2 → lock B → wait for A

👉 System freezes

✅ 14. Starvation

Thread never gets CPU/resources.

Example:

VIP customers always served first.

✅ 15. Concurrency vs Parallelism

Concurrency → many tasks at once (not necessarily simultaneous)
Parallelism → tasks truly run at same time (multiple cores)

✅ 16–17. volatile vs synchronized
volatile

❌ No atomicity
❌ No locking

synchronized

✔ Visibility

✔ Mutual exclusion
✔ Full thread safety

👉 volatile ≠ replacement for synchronized
